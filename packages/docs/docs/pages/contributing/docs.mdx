# Contributing to Documentation

This guide explains how to contribute to the Claude Swarm documentation, including writing, editing, and maintaining the docs site built with [Vocs](https://vocs.dev).

:::tip[Documentation-Driven Development]
Claude Swarm follows a **documentation-first approach**. We write docs before code to ensure clear requirements and better user experience.
:::

## Writing Guidelines

### Content Standards

#### Use Clear, Actionable Language

```markdown
{/* ‚úÖ Good: Clear and actionable */}
## Set up GitHub Integration

Configure your GitHub token to enable issue tracking:

```bash
claude-swarm config set github.token YOUR_TOKEN
```

{/* ‚ùå Bad: Vague and unclear */}
{/* ## GitHub

You might want to configure GitHub stuff. */}

#### Include Code Examples

Always provide working code examples:

```typescript
// ‚úÖ Complete, runnable example
import { WorktreeManager } from '@claude-swarm/core'

const manager = new WorktreeManager('./project')

await manager.createWorktree({
  path: './worktrees/feature-123',
  branch: 'feature-123',
  agentId: 'feature-dev'
})

// ‚ùå Incomplete fragment

// ... do something
```

#### Use Appropriate Callouts

```markdown
{/* For important warnings */}
:::warning[Important]
Always backup your work before running cleanup operations.
:::

{/* For helpful tips */}
:::tip[Pro Tip]
Use agent IDs to organize different types of work.
:::

{/* For critical information */}
:::danger[Critical]
Never run `--force` without checking active worktrees first.
:::

{/* For additional information */}
:::info[Learn More]
Check out the [API Reference](/api) for detailed function signatures.
:::
```

## Local Development

### Setting Up the Docs Site

```bash
# Navigate to docs package
cd packages/docs

# Install dependencies
bun install

# Start development server
bun run dev

# The site will be available at http://localhost:5173
```

### Available Scripts

```bash
# Development
bun run dev          # Start dev server with hot reload

# Building
bun run build        # Build for production
bun run preview      # Preview production build

# Linting (from root)
bun run lint         # Lint all code
bun run typecheck    # Type checking
```

### Development Workflow

1. **Create a new branch** for your documentation changes
2. **Make your changes** in the appropriate `.mdx` files
3. **Test locally** using `bun run dev`
4. **Check the build** with `bun run build`
5. **Submit a pull request** with clear description

## Configuration

### Site Configuration (`vocs.config.ts`)

The main configuration file controls site behavior:

```typescript
export default defineConfig({
  // Basic metadata
  title: 'Claude Swarm',
  description: 'Multi-agent development environment',
  
  // Navigation structure
  sidebar: [
    {
      text: 'Getting Started',
      items: [
        { text: 'Introduction', link: '/' },
        { text: 'Quick Start', link: '/getting-started' }
      ]
    }
  ],
  
  // Theme and branding
  theme: {
    accentColor: { light: '#3b82f6', dark: '#60a5fa' }
  },
  
  // OG image configuration
  ogImageUrl: 'https://vocs.dev/api/og?logo=%logo&title=%title',
  
  // Search configuration
  search: {
    boostDocument(documentId) {
      return documentId.includes('/concepts/') ? 2 : 1
    }
  }
})
```

### Adding New Sections

To add a new documentation section:

1. **Create the directory structure**:
   ```bash
   mkdir -p docs/new-section
   ```

2. **Add to sidebar configuration**:
   ```typescript
   sidebar: [
     // ... existing sections
     {
       text: 'New Section',
       items: [
         { text: 'Overview', link: '/new-section/' },
         { text: 'Advanced', link: '/new-section/advanced' }
       ]
     }
   ]
   ```

3. **Create content files**:
   ```bash
   touch docs/new-section/index.mdx
   touch docs/new-section/advanced.mdx
   ```

## Style Guide

### Writing Style

- **Use active voice**: "Run the command" instead of "The command should be run"
- **Be concise**: Remove unnecessary words and filler
- **Use consistent terminology**: Stick to established terms (worktree, not work-tree)
- **Write scannable content**: Use headers, lists, and callouts effectively

### Code Style

- **Include complete examples**: Show full working code, not fragments
- **Add comments for clarity**: Explain non-obvious parts
- **Use TypeScript**: Show types for better understanding
- **Test your examples**: Ensure all code examples actually work

### Visual Style

- **Use emojis sparingly**: Only in callouts and section headers where they add clarity
- **Consistent formatting**: Follow established patterns for similar content
- **Proper hierarchy**: Use heading levels logically (H1 ‚Üí H2 ‚Üí H3)

### Search Optimization

Configure search behavior to boost important content:

```typescript
search: {
  boostDocument(documentId) {
    // Boost getting started and concepts
    if (documentId.includes('/getting-started') || documentId.includes('/concepts/')) {
      return 2
    }
    // Reduce API reference prominence
    if (documentId.includes('/api/')) {
      return 0.5
    }
    return 1
  }
}
```

## Review Process

### Before Submitting

1. **‚úÖ Check spelling and grammar**
2. **‚úÖ Verify all links work**
3. **‚úÖ Test code examples**
4. **‚úÖ Build the site locally** (`bun run build`)
5. **‚úÖ Check responsive design**

### Pull Request Guidelines

When submitting documentation changes:

1. **Clear title**: "docs: add worktree management guide"
2. **Describe changes**: What was added/changed and why
3. **Include screenshots**: For visual changes
4. **Link related issues**: Reference GitHub issues if applicable

### Review Checklist

Reviewers should check:

- [ ] Content is accurate and up-to-date
- [ ] Code examples work as expected  
- [ ] Navigation and links function properly
- [ ] Formatting is consistent
- [ ] No spelling or grammar errors
- [ ] Site builds without errors

## Deployment

The documentation site is automatically deployed when changes are merged to the main branch. The deployment process:

1. **Build the site** using `bun run build`
2. **Deploy to production** hosting
3. **Update search index** for new content
4. **Generate new OG images** for changed pages
